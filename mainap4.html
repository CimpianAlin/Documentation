D.2 Installing the Framework from Source

In order to install the Core Framework from source, the redhawk-src-1.9.0.tar.gz must be downloaded.

Installing from source provides the advantage of allowing the user to set a variety of compile-time configuration options, including:

  • Disable log4cxx support
  • --disable-log4cxx
  • Enable LOG_TRACE statements
  • --enable-trace
  • Enable persistence support:
  • --enable-persistence=<persist_type>
  • (Support types: bdb, gdbm, sqlite, none)
  • Disable framework java support:
  • --disable-java
  • Change the ossie root directory
  • --with-ossie=<dir>
  • Change the SDR root directory
  • --with-sdr=<dir>

D.3 Setting Environment Variables

The environment variables SDRROOT and OSSIEHOME must be set in order to run REDHAWK. OSSIEHOME points to the location of installed libraries, header files, and executables. SDRROOT is the installation target of user-developed resources (e.g., Components and Waveforms), which implies that the developer must have write permissions to SDRROOT and its subdirectories. D.4 Modifying the Python parsers

The files dcd.py, dmd.py, dpd.py, prf.py, profile.py, sad.py, scd.py, and spd.py are generated with a tool called generateDS. If the xsd is modified, then these files must be re-generated. Unlike in C++, this process is not tied into autoconf, so the integration process of the newly generated files is manual (generate the new Python files and over-write the existing ones). Furthermore, due to some aspects of the generated code, namely that some attributes are mapped to Python reserved words along with other minor issues, the generated files must be hand-modified to work with REDHAWK. These are the modifications that must be made:

  • Change every constructor attribute id to id_
  • Change every member variable id to id_
  • Change the class range to range_. Make sure to also update the name where the class is instantiated
