Chapter 10
The Runtime Environment in Brief

10.1 Introduction

The REDHAWK runtime environment provides a mechanism for managing the life cycle (creation/teardown/initialization) of Components as well as the interconnection of deployed Components. A full discussion of the REDHAWK runtime environment is outside the scope of this introductory tutorial but can be found in Chapter 11.

The primary goal of the runtime environment is to support the necessary infrastructure to deploy and manage interconnected Components as a running Application.

The runtime environment is personified by two binaries, a Domain Manager program and a Device Manager program. The Domain Manager program hosts an instance of a DomainManager class as well as supporting objects (Application Factory, Application, File Manager). The Device Manager program hosts an instance of a DeviceManager object and an instance of a File System. The only reason why the major classes making up these programs are imported is because their API is remotely available, so a user can arbitrarily interact with these objects. Irrespective of their API, these objects will exclusively reside in either the Domain Manager or Device Manager programs.

A single REDHAWK system instance will have one Domain Manager and an arbitrary number of Device Managers. The Domain Manager’s role is as a central bookkeeper as well as a single point where Applications can be deployed or torn down. A Device Manager is present in each host in the REDHAWK network area; in a purely processing context, there would be a single Device Manager in each computer that is intended to host running Components. Each Device Manager will have associated with it a set of Devices that act as proxies for whatever hardware is running on the computer; in a purely processing context, there is a single Device proxy describing the microprocessor for each hardware platform Device Manager.

The Domain Manager program:

  • receives an XML file describing a Waveform that is to be deployed.
  • scans all running Devices on all Device Managers for a suitable place to deploy the Components making up the Waveform.
  • uses the File Manager / File System to copy whatever files are necessary to run the Components to the target Device Managers.
  • remotely invokes the Component processes.
  • interconnects Components over the network.
  • tears down Applications appropriately.

10.1.1 Launching a Domain

If REDHAWK was installed from RPM, a Domain Manager and Device Manager are ready for immediate use on the localhost. To run using a default Domain in the IDE, follow these steps:

  1. In the SCA Explorer view (by default on the right-side of the window) right-click the Target SDR element and select Launch..., as seen in Figure 10.1
  2. In the Device Manager section, select DevMgr_hostname, as seen in Figure 10.2
  3. Select OK


Figure 10.1: Right-click Target SDR from the SCA Explorer and select Launch...


Figure 10.2: Select Device Manager on the Launch selection

This will launch both a Domain Manager and a Device Manager that manages a single GPP Device. The output from both the Domain Manager and Device Manager will be visible in the Console view. If this view is not visible select Window > Show View > Console. These processes can be stopped using the red square stop button. One can switch between the Domain Manager and Device Manager consoles using the console switcher button (it looks like a small computer monitor with an arrow next to it), as seen in Figure 10.3.


Figure 10.3: Location of console switcher

A Domain connection called REDHAWK_DEV in the SCA Explorer view should be visible, as seen in Figure 10.4. Its state should be CONNECTED and there should be no errors. A Domain Manager process and a Device Manager process should now exist on the host.


Figure 10.4: Domain connections shown on IDE

10.1.2 Shutting Down the Domain

Under normal circumstances, the Domain Manager and Device Manager would remain running indefinitely; these programs are designed to remain running for extended periods of time as different parts of the overall Domain (e.g., Device Managers, Applications, and files on $SDRROOT) come and go. However, for the purpose of this tutorial, the process for shutting down a running Domain is discussed. To cleanly shutdown, it is best to disconnect the Domain and stop the processes that have been started.

  1. In the SCA Explorer view right-click the REDHAWK_DEV Domain and select Disconnect, as seen in Figure 10.5
  2. In the Console view, select the Device Manager Console from the console button (looks like a computer monitor), as seen in Figure 10.3.
  3. Click the red square to stop the Device Manager, as seen in Figure 10.6.
  4. In the Console view, select the Domain Manager console from the console button (looks like a computer monitor), as seen in Figure 10.3.
  5. Click the red square to stop the Domain Manager
  6. Select File > Exit


Figure 10.5: Right-click REDHAWK_DEV from the SCA Explorer and select Disconnect


Figure 10.6: Red Square stop button

The Domain Manager and Device Manager processes should no longer exist on the host. The Domain entry will remain in the SCA Explorer view with a DISCONNECTED state indicating that the Domain is no longer visible. This decoupling of the running Domain from the environment is there to allow the SCA Explorer to interact with an arbitrary number of Domains on a network where each Domain’s life cycle is outside the control of the IDE.
